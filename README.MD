# AGENT.md

## 1. Назначение проекта

Это внутреннее веб-приложение для **учета предложений / требований** по развитию и рефакторингу системы чрезвычайных происшествий.

Система заменяет неструктурированную работу с Excel-матрицей требований и должна обеспечивать:

- централизованный учет предложений;
- фильтрацию, поиск, просмотр и редактирование карточек;
- обсуждение внутри карточки;
- импорт и экспорт Excel;
- разграничение прав доступа;
- простую корпоративную авторизацию.

---

## 2. Бизнес-контекст

В системе фиксируются предложения от разных участников процесса:

- ДИТ
- Система 112
- Система 101
- Танто-С

Каждое предложение содержит данные из матрицы требований: краткое наименование, инициатор, ответственный, очередь, статус, текст предложения, описание проблем, обсуждение, пункт ТЗ, примечание, ГК и т.д.

## 4. Ключевые роли и доступ

## 4.1. Типы пользователей

В системе есть пользователи с такими полями:
- ФИО
- организация
- email
- пароль
- accessLevel
- isSuperuser
- isActive

### Допустимые организации
- `ДИТ`
- `112`
- `101`
- `Танто-С`

### Допустимые уровни доступа
- `read` — только чтение
- `edit` — чтение и редактирование

### Суперпользователь
Суперпользователь:
- видит страницу справочника пользователей;
- может создавать и редактировать пользователей;
- не имеет права менять пароль другому пользователю;
- меняет пароль только себе через профиль.

---

## 4.2. Правила доступа

### Пользователь `read`
Может:
- войти в систему;
- открыть таблицу;
- фильтровать и искать;
- открывать карточки предложений;
- просматривать карточки;
- экспортировать Excel;
- открыть профиль;
- сменить свой пароль;
- выйти.

Не может:
- создавать предложения;
- редактировать предложения;
- сохранять изменения;
- писать комментарии;
- архивировать / восстанавливать;
- импортировать Excel;
- видеть шаблоны импорта;
- видеть инпуты редактирования в карточке.

### Пользователь `edit`
Может:
- всё, что может `read`;
- создавать предложения;
- редактировать предложения;
- сохранять изменения;
- писать комментарии;
- архивировать и восстанавливать;
- импортировать Excel;
- использовать шаблоны.

### Суперпользователь
Может:
- всё, что может `edit`;
- открывать справочник пользователей;
- создавать пользователей;
- редактировать пользователей.

Не может:
- менять пароль другому пользователю.

## 5. Ключевые бизнес-правила

## 5.1. Предложения / требования

Основная сущность — `Requirement`.

Ключевые поля:
- `taskIdentifier`
- `shortName`
- `initiator`
- `responsiblePerson`
- `sectionName`
- `proposalText`
- `problemComment`
- `discussionSummary`
- `implementationQueue`
- `contractName`
- `contractTZFunctionId` — (опционально) ссылка на запись функции ТЗ из справочника ГК
- `noteText`
- `tzPointText`
- `statusText`
- `systemType`
- `authorName`
- `authorOrg`
- `createdAt`
- `updatedAt`
- `lastEditedBy`
- `lastEditedOrg`
- `isArchived`
- `archivedAt`
- `archivedBy`
- `archivedByOrg`

## 5.2. Идентификатор задачи

Идентификатор задачи формируется автоматически в логике системы по очереди и порядковому номеру, в стиле:

- `ПОВ1.11`
- `ПОВ2.19`

Где:
- первая цифра — номер очереди;
- вторая часть — порядковый номер внутри этой очереди.

Если меняется логика генерации `taskIdentifier`, это нужно делать аккуратно и не ломать существующие данные.

## 5.3. Очереди

В системе есть справочник очередей.

По умолчанию:
- `1 очередь`
- `2 очередь`
- `3 очередь`

Пользователь с правом редактирования может использовать существующие очереди.
Новые очереди могут добавляться через справочник / API.

Цвета очередей в UI:
- `1 очередь` — пастельно-розовый
- `2 очередь` — пастельно-жёлтый
- `3 очередь` — пастельно-зелёный

## 5.4. Статусы

Базовые статусы:
- `Новое`
- `Учтено`
- `Выполнено`

Разрешён и произвольный пользовательский статус.
Если пользователь вводит свой статус, он сохраняется как строка.

## 5.5. ГК / контракт

У предложения есть поле `contractName`.

Сущность в БД — `ContractDictionary` (справочник ГК).
Дополнительно у ГК есть:
- `description` — главная информация / описание (текст);
- связанные **этапы** (`ContractStage`): номер этапа (1, 2, 3, …) и отображаемое имя;
- связанные **функции ТЗ** (`ContractTZFunction`): наименование функции, этап, **номер по НМЦК** (строка, может быть вида `1.1.2`), номер раздела по ТЗ;
- **вложения** файлов ТЗ и НМЦК (`ContractAttachment`, несколько файлов на ГК, типы `tz` / `nmck`).

Поведение для поля ГК в предложении:
- выбор из существующих значений справочника (и каскад «этап → функция» в UI при необходимости);
- при создании предложения через форму и при сохранении связи с функцией backend может выставлять `tzPointText` по выбранной функции;
- при импорте предложений из Excel по-прежнему возможен ввод/создание ГК по тексту (как раньше — через поле ГК в шаблоне).

Отдельная страница **«Справочник ГК»** (`/gk-directory`): управление карточкой ГК, этапами, функциями ТЗ, Excel-импортом функций и загрузкой файлов ТЗ/НМЦК.

## 5.6. Пункт ТЗ

У предложения есть поле `tzPointText`.

Источники значения:
- текст вручную или из импорта Excel (как раньше);
- при передаче `contractTZFunctionId` при создании/обновлении предложения backend может **сформировать** `tzPointText` из выбранной функции (номер раздела по ТЗ и наименование).

Глобальный справочник **пунктов ТЗ** (`TZPoint`) и endpoint импорта `POST /api/import/tz-points` в проекте сохраняются для совместимости; **кнопок «Импорт ТЗ» и «Шаблон ТЗ» на главной странице реестра нет** — основной сценарий работы с функциями ТЗ привязан к ГК и странице справочника ГК.

## 5.7. Комментарии

Каждое предложение может иметь список комментариев.

Комментарий содержит:
- текст;
- автора;
- организацию автора;
- дату создания.

Комментарии доступны для чтения всем авторизованным пользователям.
Добавлять комментарии могут только пользователи `edit` и суперпользователь.

## 5.8. Архив

Предложение можно архивировать и восстанавливать.

Поля архива:
- `isArchived`
- `archivedAt`
- `archivedBy`
- `archivedByOrg`

Архивирование доступно только пользователю `edit` и суперпользователю.

## 5.9. Справочник ГК (расширение)

Назначение:
- единое место для описания ГК, этапов реализации и набора функций ТЗ;
- загрузка **нескольких** файлов ТЗ и НМЦК на каждую ГК (допустимые расширения на backend: в т.ч. `.doc`, `.docx`, `.xls`, `.xlsx`, `.pdf`);
- импорт функций ТЗ из Excel (`POST /api/import/gk-tz-functions`).

Файлы вложений сохраняются на диске сервера (каталог `uploads/contracts/...` относительно рабочей директории backend); метаданные — в таблице вложений (`ContractAttachment`).

## 6. Аутентификация и безопасность

## 6.1. Общая схема

В системе используется JWT-аутентификация.

Пользователь входит по:
- email
- пароль

После логина backend возвращает:
- `accessToken`
- профиль пользователя

Токен хранится на frontend в `localStorage`.

## 6.2. Что должен учитывать агент

При любых изменениях аутентификации нельзя ломать:
- `Authorization: Bearer <token>`
- `/api/auth/login`
- `/api/auth/me`
- `/api/auth/change-password`

## 6.3. Правила ИБ

Обязательные ограничения:
- пароль другого пользователя нельзя менять из административной панели;
- пароль меняет только сам пользователь через профиль;
- read-only пользователь не должен видеть edit-контролы;
- отсутствие кнопки в UI не заменяет backend-проверку — backend тоже обязан проверять права;
- суперпользователь не должен получать скрытую возможность смены чужого пароля.

## 7. UI / UX правила

## 7.1. Общий стиль
- интерфейс должен быть простым;
- визуально это корпоративный реестр;
- не перегружать лишней декоративностью;
- использовать аккуратные карточки, таблицу и стандартные элементы Element Plus.

## 7.2. Шапка страницы
В шапке:
- название страницы;
- бейдж организации;
- ФИО пользователя;
- кнопки действий.

Кнопка **Профиль**:
- должна идти **перед кнопкой "Выйти"**;
- должна быть визуально выделена другим цветом;
- не должна сливаться с остальными кнопками.

## 7.3. Таблица
Таблица — основной режим работы системы.

Правила:
- таблица должна быть читаемой на широком экране;
- не должно быть пустого хвоста справа;
- последние столбцы не должны обрезаться;
- горизонтальный скролл должен относиться к таблице, а не ко всей странице;
- при необходимости используется плавающий нижний горизонтальный скролл;
- заголовки таблицы должны быть закреплены;
- read-only пользователь видит только кнопку открытия карточки;
- edit-пользователь видит кнопки открытия, архивирования, восстановления.

## 7.4. Карточка предложения
Карточка открывается по клику на строку.

### Для `read`
- только просмотр;
- никаких инпутов;
- никаких кнопок сохранения;
- никаких кнопок архива/восстановления;
- никаких форм комментариев.

### Для `edit`
- доступны инпуты;
- доступно сохранение;
- доступно добавление комментариев;
- доступно архивирование/восстановление;
- при необходимости — выбор ГК и каскад «этап → функция ТЗ» для привязки к справочнику ГК (`contractTZFunctionId`).

На главной странице реестра (`RequirementsPage.vue`) есть переход **«Справочник ГК»** на маршрут `/gk-directory`.

## 8. Backend API (актуальная логика)

## 8.1. Auth
- `POST /api/auth/login`
- `GET /api/auth/me`
- `POST /api/auth/change-password`

## 8.2. Requirements
- `GET /api/requirements`
- `GET /api/requirements/:id`
- `POST /api/requirements` — в теле JSON допускается `contractTZFunctionId` (опционально); если указан и валиден для выбранной ГК, текст `tzPointText` может быть сформирован из функции
- `PUT /api/requirements/:id` — то же правило для `contractTZFunctionId`
- `POST /api/requirements/:id/comments`
- `POST /api/requirements/:id/archive`
- `POST /api/requirements/:id/restore`

## 8.3. Export / Import
- `GET /api/export/requirements`
- `POST /api/import/requirements`
- `POST /api/import/tz-points` — глобальный импорт пунктов ТЗ (API сохранён)
- `POST /api/import/gk-tz-functions` — импорт **функций ТЗ** по ГК из Excel (edit/superuser)

## 8.4. Dictionaries
- `GET /api/authors`
- `GET /api/tz-points`
- `GET /api/contracts` — список ГК (как раньше)
- `GET /api/contracts/search` — подсказки по названию ГК
- `GET /api/contracts/:id` — детали ГК: описание, этапы, функции ТЗ
- `GET /api/contracts/:id/stages` — список этапов ГК
- `GET /api/contracts/:id/stages/:stageNumber/functions` — список функций ТЗ этапа
- `GET /api/contracts/:id/attachments` — список вложений ТЗ/НМЦК
- `GET /api/contracts/attachments/:attachmentId/download` — скачать вложение
- `GET /api/queues`
- `POST /api/queues`

## 8.4.1. ГК — изменение данных (edit / superuser)

- `POST /api/contracts` — создать/актуализировать ГК (имя, описание)
- `PUT /api/contracts/:id` — обновить ГК
- `POST /api/contracts/:id/stages` — добавить этап
- `POST /api/contracts/:id/functions` — создать/обновить функцию ТЗ
- `POST /api/contracts/:id/attachments` — загрузить файлы (`multipart/form-data`: `type` = `tz` | `nmck`, поле файлов `files`)

## 8.5. Admin
- `GET /api/admin/users`
- `POST /api/admin/users`
- `PUT /api/admin/users/:id`

Важно: маршрута смены пароля другого пользователя **быть не должно**.

## 11. Импорт / экспорт Excel

Система поддерживает:
- импорт предложений (`POST /api/import/requirements`, UI на главной странице реестра);
- импорт пунктов ТЗ в глобальный справочник (`POST /api/import/tz-points`, **без кнопки на главной** — API и данные сохраняются);
- импорт **функций ТЗ по ГК** (`POST /api/import/gk-tz-functions`, UI на странице **Справочник ГК**);
- экспорт текущего реестра в Excel;
- скачивание шаблонов: предложения; шаблон функций ТЗ для ГК (`excelTemplates.ts`).

При изменениях импорта/экспорта нельзя ломать:
- поля `contractName`
- поля очереди
- поля статуса
- поля ТЗ (`tzPointText`) и согласованность с `contractTZFunctionId`, если используется привязка к функции
- формирование идентификатора
- работу с архивными записями

## 12. Правила для изменений кода

При любых правках агент должен соблюдать следующие правила:

### 12.1. Всегда комментировать новый код
В проекте принято сопровождать важные блоки кода комментариями.
Любой новый или существенно изменённый код нужно комментировать.

### 12.2. Не ломать права доступа
Любое изменение UI или API должно сохранять:
- read-only ограничения;
- backend-проверки прав;
- запрет смены чужого пароля;
- доступ суперпользователя только к разрешённым admin-функциям.

### 12.3. Не ломать таблицу
При изменениях таблицы нельзя допускать:
- обрезание последних столбцов;
- двойные горизонтальные скроллы;
- пустой хвост справа;
- исчезновение действий из последнего столбца;
- потерю sticky-заголовков.

### 12.4. Сначала учитывать существующий UX
Перед рефакторингом нужно учитывать уже принятые решения:
- профиль расположен перед выходом;
- профиль визуально выделен;
- read-only UI строго ограничен;
- таблица — основной рабочий режим;
- карточка — вторичный режим просмотра/редактирования.

---

## 13. Что нельзя делать

Агенту нельзя:
- возвращать в систему общий пароль вместо JWT;
- добавлять смену пароля одного пользователя другим пользователем;
- показывать read-only пользователю формы редактирования;
- показывать read-only пользователю кнопки создания/импорта/архивации;
- ломать работу таблицы ради косметического рефакторинга;
- менять организации на произвольные значения вне утверждённого списка;
- добавлять неоговорённые роли доступа.

Если ты вносишь изменения в проект, проверь:

1. Не сломалась ли JWT-авторизация.

3. Не может ли кто-то менять пароль другому пользователю.

5. Не появились ли двойные горизонтальные скроллы.

AGENT.md

Главный файл документации для AI-агента.

Содержит:

правила проекта;
бизнес-логику;
ограничения;
структуру системы;
требования к безопасности и UI.
17.2. Backend
backend/

Папка backend-приложения на Go.

Содержит:

точку входа;
бизнес-логику API;
работу с БД;
модели;
middleware;
JWT-логику;
конфигурацию.
backend/cmd/app/main.go

Точка входа backend-приложения.

Отвечает за:

загрузку конфигурации;
подключение к БД;
инициализацию JWT;
регистрацию маршрутов;
CORS;
запуск HTTP-сервера;
раздачу frontend в режиме single-service deploy.
backend/internal/config/config.go

Загрузка и хранение конфигурации приложения.

Отвечает за:

чтение переменных окружения;
параметры БД;
JWT secret;
frontend origin;
стартовые параметры суперпользователя.
backend/internal/db/postgres.go

Подключение к PostgreSQL и миграции.

Отвечает за:

создание подключения к базе;
AutoMigrate;
сидирование стандартных очередей;
сидирование суперпользователя.
backend/internal/models/models.go

Описание моделей базы данных.

Содержит сущности:

Requirement (в т.ч. опционально `contractTZFunctionId`)
Comment
TZPoint
AuthorDictionary
QueueDictionary
ContractDictionary (ГК; поле `description`)
ContractStage (этапы ГК)
ContractTZFunction (функции ТЗ; номер по НМЦК — строка)
ContractAttachment (файлы ТЗ/НМЦК, привязанные к ГК)
User
backend/internal/security/jwt.go

Логика создания и проверки JWT.

Отвечает за:

генерацию access token;
парсинг access token;
описание claims.
backend/internal/middleware/auth_middleware.go

Middleware авторизации и проверки прав.

Отвечает за:

проверку Bearer token;
загрузку пользователя из БД;
помещение данных пользователя в gin context;
ограничение доступа:
только авторизованный;
только edit/superuser;
только superuser.
backend/internal/handlers/auth_handler.go

Handler аутентификации и профиля.

Отвечает за:

логин;
получение текущего профиля;
смену собственного пароля.
backend/internal/handlers/user_handler.go

Handler административной работы с пользователями.

Отвечает за:

список пользователей;
создание пользователя;
редактирование пользователя.

Важно:

не должен содержать смену пароля другого пользователя.
backend/internal/handlers/requirement_handler.go

Основной handler работы с предложениями.

Отвечает за:

список предложений;
получение карточки;
создание предложения;
обновление предложения;
добавление комментариев;
архивирование;
восстановление;
экспорт в Excel;
импорт предложений.
backend/internal/handlers/dictionary_handler.go

Handler справочников.

Отвечает за:

поиск авторов;
поиск пунктов ТЗ;
чтение и поиск ГК (базовый список);
чтение очередей;
создание очередей;
импорт ТЗ (глобальные пункты).
backend/internal/handlers/contract_directory_handler.go

Расширенный справочник ГК.

Отвечает за:

детали ГК (этапы, функции ТЗ);
создание/обновление ГК, этапов, функций;
импорт функций ТЗ из Excel;
загрузку и выдачу вложений ТЗ/НМЦК.
backend/go.mod

Описание зависимостей backend-проекта.

backend/go.sum

Контрольные суммы Go-зависимостей.

backend/.env

Локальные переменные окружения backend.

Содержит:

параметры БД;
JWT secret;
origin фронтенда;
данные стартового суперпользователя.

17.6. Frontend components
frontend/src/components/RequirementFormDialog.vue

Диалог создания нового предложения.

Используется только пользователями с правом редактирования.

Может включать выбор ГК и каскад «этап → функция ТЗ» (`contractTZFunctionId`).

frontend/src/components/RequirementDetailsDrawer.vue

Drawer карточки предложения.

Поддерживает два режима:

read-only просмотр;
редактирование для edit/superuser.

Содержит:

основные поля карточки;
при редактировании — связь с функцией ТЗ через справочник ГК (по возможности);
комментарии;
добавление комментария;
архивирование/восстановление для edit.
frontend/src/components/ImportExcelDialog.vue

Диалог импорта Excel.

Используется для:

импорта предложений;
импорта пунктов ТЗ (если вызван из кода);
импорта функций ТЗ для ГК (`gkFunctions`).

Должен быть скрыт для read-only пользователя.

frontend/src/components/GKContractDialog.vue / GKFunctionDialog.vue

Модальные окна справочника ГК (карточка ГК и функция ТЗ).

frontend/src/components/ProfileDrawer.vue

Drawer профиля пользователя.

Содержит:

ФИО;
почту;
организацию;

frontend/src/components/AdminUserDialog.vue

Модальное окно создания/редактирования пользователя.

Используется только в административной странице.

Визуальный тег очереди.

Изменения в этих файлах могут сломать:

отображение read-only режима;
таблицу;
карточку предложения;
безопасность системы.
